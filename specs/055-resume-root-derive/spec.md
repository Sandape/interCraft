# Feature Specification: 简历中心提升 — 根简历、派生简历、一键派生与 AI 优化建议

**Feature Branch**: `055-resume-root-derive`

**Created**: 2026-07-09

**Status**: Draft

**Requirement ID**: REQ-055

**Input**: User description: "简历中心提升：根简历、派生简历、一键派生与 AI 优化建议。产品交互设计为主，将简历中心升级为长期职业素材库 + 岗位定向派生简历 + AI 优化建议 + 求职追踪联动的智能简历系统。"

## Background

当前产品已具备「简历中心」与「求职追踪」能力，但简历中心仍偏传统管理与编辑：用户难以在投递时快速筛选匹配素材；不同 JD 需反复手改；单份简历上下文不足导致 AI 优化泛化甚至编造；求职追踪中的岗位与 JD 尚未充分反哺简历生成。

REQ-055 将简历中心升级为：

1. **根简历**：长期、可很长的职业素材库（非投递稿）
2. **派生简历**：面向具体岗位 JD、严格页数约束的独立投递快照
3. **一键派生**：从求职追踪选岗 → 解析 JD → 从根简历取材 → 生成并校准页数
4. **简历助手 Agent**：解析、筛选、重组、页数控制、优化建议、补充追问，且禁止编造事实
5. **闭环**：派生简历绑定岗位，可在求职追踪中回看投递版本

### 产品模块

- 求职者帮助平台 — 简历中心
- 求职追踪（岗位 / JD 来源与回链）
- 简历助手 Agent（AI 能力载体）

### 设计原则（产品层）

1. **素材库与投递稿分离**：根简历求全，派生简历求准、求页数可控。
2. **事实优先于匹配**：匹配度再高也不能编造；无依据只能建议补充。
3. **页数是硬约束**：目标页数 1/2/3 页，最终导出 PDF 必须严格等于目标页数。
4. **快照独立**：派生简历不随根简历自动同步。
5. **可解释**：用户能看到「为什么这样写 / 删 / 强化」。
6. **本阶段以交互与产品逻辑为主**：不展开接口、库表、模型选型与编排实现。

## Competitive Research

### 1. Teal

**启发**：Job Tracker + Resume Builder + Match Score 一体化；按岗定制简历；JD 定向 audit / rewrite。

**可借鉴**：

1. 求职追踪与简历生成强绑定
2. 每岗可生成定向简历
3. 提供 JD 匹配反馈（MVP 以清单式分析为主，分数可后置）
4. 优化建议围绕目标岗位，而非泛泛润色

### 2. Rezi

**启发**：AI Keyword Targeting；展示已覆盖 / 缺失关键词；引导补 bullet。

**可借鉴**：

1. 关键词分已覆盖、弱覆盖、未覆盖
2. 缺失关键词不得硬塞，须有根简历事实依据
3. 建议落到具体模块 / bullet / 位置
4. 用完成度 / 进度感提升优化动力（非强制评分）

### 3. Jobscan / Resume Worded

**启发**：ATS 匹配扫描；missing keywords、格式问题、match gaps。

**可借鉴**：

1. 同时服务 ATS 解析与招聘者阅读
2. 识别格式风险（复杂表格、图片化文本、不可解析模块等）
3. 建议分类：内容 / 关键词 / 格式 / 风险
4. 不以 ATS 分数为唯一目标，避免关键词堆砌

### 4. Huntr / Careerflow

**启发**：求职工作台；Tracker、简历、求职信、面试准备同流程。

**可借鉴**：

1. 简历是投递与面试准备的上游资产
2. 派生简历沉淀在岗位维度，便于后续求职信 / 面试材料
3. 投递反馈可反向影响优化策略（后续迭代）
4. 用岗位状态管理多份派生简历

### 5. Enhancv / Kickresume

**启发**：AI builder + 模板 + JD 定向生成；视觉与导出一体。

**可借鉴**：

1. AI 生成与模板、页面渲染联动，不只出文本
2. 生成前选择目标页数与模板
3. 编辑器内 AI 面板（右侧建议 / 匹配 / 页数）
4. 输出兼顾 PDF 渲染与 ATS 友好

### 6. 本产品差异化

竞品多围绕「当前简历 + JD」优化；本产品强调：

1. 根简历是长期职业素材库，可很长，不受投递页数限制
2. 派生简历是岗位 JD 定向独立快照
3. AI 只能从根简历与用户确认补充中取材，降低幻觉
4. 求职追踪岗位 JD 直接驱动一键派生
5. 每份派生绑定岗位、JD、根简历版本与生成依据
6. 后续可基于投递结果反向优化根简历与派生策略

## Core Concepts

### 根简历（Root Resume）

用户长期维护的职业素材库，**不是**最终投递简历。可非常长，不限制 1/2/3 页。目标是完整沉淀可写入简历的真实素材。

可包含：个人信息、教育、工作、项目、技能栈、工具能力、业务能力、证书/奖项、作品集/链接、项目成果、可量化数据、多岗位方向可复用补充素材、用户补充的项目细节（背景、职责、难点、结果）等。

鼓励结构化维护，但不强制一开始完全结构化；支持「模块化编辑 + AI 辅助整理」。

### 派生简历（Derived Resume）

基于根简历、目标 JD、用户选择的目标页数生成的岗位定向投递版本。

必须：绑定岗位记录与该岗位 JD；记录生成时根简历版本；生成后独立快照保存；不因根简历后续修改自动变化。

可：手动编辑；同岗重新生成新版本；导出（MVP：PDF；Word 后续）；查看与 JD 的匹配分析；查看 AI 内容来源依据。

### 一键派生（One-Click Derive）

从求职追踪选岗 → 读 JD → 从根简历筛选/重组/压缩/排序/优化 → 严格按目标页数生成派生简历。

### 简历助手 Agent

产品侧 AI 能力载体。职责：解析 JD、判断能力要求、检索根简历素材、决定取舍与排序、生成符合页数的派生稿、提供编辑器优化建议、对缺失信息提问、标注来源、避免编造。**不是**随意生成简历文案的工具。

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 维护根简历 (Priority: P1)

作为求职者，我希望把项目、技能、经历和成果统一沉淀到根简历，这样每次投递不必重新回忆和整理素材。

**Why this priority**: 根简历是一键派生与 AI 优化的唯一事实来源；没有可用根简历，后续能力无法成立。

**Independent Test**: 无根简历用户可创建并编辑；有上传/旧简历可导入为初稿；根简历可很长且不强制页数；可查看素材完整度提示（非强制分）。

**Acceptance Scenarios**:

1. **Given** 用户进入简历中心且无根简历，**When** 看到根简历空状态，**Then** 可一键创建空白根简历或从已有上传简历解析为初稿。
2. **Given** 用户已有多份旧简历，**When** 选择「AI 合并为根简历草稿」，**Then** 系统生成可编辑草稿，且明确标注为草稿待确认，不直接覆盖已有根简历正文（若已有则先预览合并结果）。
3. **Given** 用户在根简历编辑页，**When** 使用模块化编辑、自由文本补充、打标签、关联技能与项目，**Then** 内容可保存，且不出现 1/2/3 页限制提示。
4. **Given** 某项目缺少背景/职责/方案/结果/量化数据，**When** 查看完整度提示，**Then** 展示缺失维度清单，但不阻止保存或派生。
5. **Given** 用户在派生流程中补充了事实并选择同步回根简历，**When** 确认同步，**Then** 根简历出现对应素材，并保留来源为「派生补充」。

---

### User Story 2 — 基于岗位一键生成派生简历 (Priority: P1)

作为求职者，我希望从求职追踪中选择已登记岗位，按该岗位 JD 自动生成针对性简历，以快速完成定向投递准备。

**Why this priority**: 一键派生是本需求核心价值闭环（追踪 → 简历）。

**Independent Test**: 有根简历 + 有 JD 的岗位 → 完成一键派生 → 得到绑定该岗位的派生快照并可预览/编辑。

**Acceptance Scenarios**:

1. **Given** 用户在简历中心 / 派生列表 / 岗位详情 / 空状态，**When** 点击「一键派生」，**Then** 展示求职追踪岗位列表（公司、岗位、状态、JD 是否完整、是否已有派生、最近更新、是否推荐生成）。
2. **Given** 所选岗位无 JD 或 JD 为空，**When** 尝试继续，**Then** 阻止派生并引导补充 JD。
3. **Given** 用户选择有 JD 的岗位，**When** 进入下一步，**Then** 展示 JD 摘要（岗位名、公司、核心要求、关键词）并要求选择目标页数与模板。
4. **Given** 用户点击「开始生成」，**When** 生成流程完成，**Then** 得到派生简历初稿预览，且记录绑定岗位、JD、根简历版本、生成时间。
5. **Given** 同岗位已有派生简历，**When** 再次一键派生，**Then** 生成新版本，不覆盖旧版本。

---

### User Story 3 — 控制简历页数 (Priority: P1)

作为求职者，我希望在生成前选择 1/2/3 页，且最终 PDF 页数严格等于选择，以满足不同渠道要求。

**Why this priority**: 「严格等于目标页数」是硬验收项，也是差异化承诺。

**Independent Test**: 分别选 1/2/3 页完成派生并导出；实测 PDF 页数等于目标；不满足时不可进入最终导出态。

**Acceptance Scenarios**:

1. **Given** 用户进入页数与模板选择步，**When** 未选目标页数，**Then** 无法开始生成。
2. **Given** 用户选择目标页数为 N（N∈{1,2,3}），**When** 系统完成渲染与自动调整，**Then** 预览与导出 PDF 的实际页数均为 N。
3. **Given** 自动调整多轮后仍无法严格等于 N，**When** 进入人工调整引导态，**Then** 给出可执行建议（换紧凑模板、隐藏模块、减项目、补素材、改选页数），且在页数未达标前禁用最终导出。
4. **Given** 用户在编辑器中手动修改导致页数偏离，**When** 尝试导出，**Then** 系统拦截并提示先恢复页数控制（自动再校准或按引导调整）。

---

### User Story 4 — 查看 AI 为什么这么写 (Priority: P2)

作为求职者，我希望知道为何前置某项目、删减某内容、强化某技能，以判断是否适合目标岗位。

**Why this priority**: 可解释性建立信任，支撑「不编造」产品承诺。

**Independent Test**: 打开任意派生简历的「生成依据 / 取舍说明」，可见模块级与关键 bullet 级解释。

**Acceptance Scenarios**:

1. **Given** 派生简历已生成，**When** 用户打开取舍说明，**Then** 可见：采用了哪些根简历素材、隐藏/压缩了哪些、排序依据与 JD 高优先级能力的对应关系。
2. **Given** 某条 bullet 被强化，**When** 用户查看依据，**Then** 可看到对应 JD 要求点与根简历来源摘要（MVP 至少到模块/项目级；细粒度证据链可后置）。
3. **Given** 某 JD 高优先级能力在根简历无证据，**When** 查看说明，**Then** 明确标注「未写入正文，已生成补充建议」，而非伪装为已具备。

---

### User Story 5 — 获得 AI 优化建议 (Priority: P2)

作为求职者，我希望在编辑派生简历时看到与 JD 的差距（关键词、数据、表达、页数等），以便继续完善。

**Why this priority**: 派生初稿后的持续优化是投递质量关键路径。

**Independent Test**: 打开派生编辑器右侧 AI 面板，可见分级建议，可预览后采纳/忽略/稍后处理。

**Acceptance Scenarios**:

1. **Given** 用户打开派生编辑器，**When** AI 建议面板加载完成，**Then** 每条建议含：优先级、类型、对应位置、问题说明、对匹配的影响、依据来源、是否可直接采纳 / 需补充 / 不建议写入。
2. **Given** 建议为「可直接采纳」，**When** 用户预览并确认，**Then** 仅在确认后写入派生简历，且不覆盖用户未确认的手动改动。
3. **Given** 建议涉及关键词补齐但根简历无依据，**When** 展示该建议，**Then** 归类为「需要补充事实」或「不建议写入」，不得一键写入虚假能力表述。

---

### User Story 6 — 补充缺失事实 (Priority: P2)

作为求职者，当根简历缺少关键数据时，我希望 AI 提出具体问题而非编造，以保持可信。

**Why this priority**: 反幻觉闭环；把「缺口」转化为可沉淀素材。

**Independent Test**: 对无证据的 JD 高优先级能力触发补充问题 → 用户填写 → 可选同步根简历 → 局部优化派生稿。

**Acceptance Scenarios**:

1. **Given** JD 强调某能力且根简历证据不足，**When** Agent 识别缺口，**Then** 生成具体补充问题（非笼统「请完善简历」）。
2. **Given** 用户提交补充信息，**When** 系统询问同步范围，**Then** 可选：仅当前派生 / 同步回根简历 / 暂不使用。
3. **Given** 用户选择同步回根简历，**When** 确认，**Then** 根简历更新且派生可基于新事实局部优化；补充来源被记录。
4. **Given** 用户尚未确认的补充内容，**When** 尝试导出，**Then** 待确认内容不得进入正式导出版（须确认或移除）。

---

### User Story 7 — 管理多个岗位简历版本 (Priority: P2)

作为求职者，我希望每个岗位保存对应派生简历，并在求职追踪中回看投递用了哪一版。

**Why this priority**: 完成「追踪 ↔ 简历」闭环，支撑多岗并行求职。

**Independent Test**: 同用户多岗位各有派生；岗位详情可见绑定列表；可查看/编辑/复制/重新生成/导出/删除。

**Acceptance Scenarios**:

1. **Given** 用户完成某岗派生，**When** 打开该岗位详情，**Then** 可见绑定的派生简历列表（名称、目标/实际页数、生成与编辑时间、是否已导出、是否待处理建议、是否基于最新根简历）。
2. **Given** 根简历已更新，**When** 用户查看旧派生，**Then** 提示「是否基于最新根简历重新生成」；选择后生成新版本。
3. **Given** 用户删除某派生版本，**When** 确认删除，**Then** 该版本从列表移除，不影响根简历与其他岗位派生。

---

### Edge Cases

1. **无根简历**：一键派生入口引导先创建/导入根简历。
2. **根简历为空**：允许进入编辑，但派生前拦截并说明素材不足。
3. **根简历很长**：派生时按 JD 优先级筛选；未采用素材进入「未采用素材」区，不静默丢弃。
4. **无求职追踪岗位**：一键派生引导先去求职追踪登记岗位，或提供快捷创建岗位+JD 后再派生。
5. **岗位无 JD**：禁止派生，引导补充。
6. **JD 过短 / 质量差**：允许尝试解析，但提示质量风险；匹配分析与建议置信度降级；必要时要求用户补充关键要求。
7. **无匹配素材**：可生成极简结构稿 + 大量「建议补充」，不得编造经历填满页数；若无法满足目标页数则进入引导态。
8. **有素材但证据不足**：正文只写有依据部分；缺口进补充问题与「不建议写入」。
9. **选 1 页但内容过多**：执行超页压缩策略；仍失败则人工引导。
10. **选 3 页但素材不足**：执行不足扩展（仅基于已有事实扩写表达/展开模块）；仍不足则引导补根简历或改选页数。
11. **模板导致页数无法严格匹配**：建议切换模板或调整版式参数（在可读性底线内）。
12. **生成含待确认事实**：标记待确认，禁止进入最终导出态直至处理。
13. **手动编辑导致页数变化**：导出前再检测；偏离则拦截。
14. **导出前页数 ≠ 目标**：禁止最终导出。
15. **重复生成同岗派生**：始终新版本，保留历史。
16. **根简历更新后旧派生**：提示可重新生成，不自动改旧稿。
17. **采纳建议会改变事实含义**：系统应警示并要求确认；若检测为新增事实则转为「需补充/确认」路径。
18. **AI 建议与用户手动内容冲突**：以用户已保存手动内容为准；建议需预览确认，默认不覆盖。
19. **网络或生成任务失败**：展示失败态与可重试；不留下半成品「可导出」状态；可保留失败任务说明。
20. **生成中途离开**：任务可后台完成并在列表提示；或明确取消策略（产品默认：允许后台完成后通知，用户可从列表进入）。

## Requirements *(mandatory)*

### Functional Requirements

#### 根简历

- **FR-001**: 系统 MUST 在简历中心首页提供与派生列表区分的「根简历」入口卡片。
- **FR-002**: 用户 MUST 能够创建、查看、编辑根简历；根简历 MUST NOT 受 1/2/3 页投递限制。
- **FR-003**: 系统 MUST 支持将已上传简历解析为根简历初稿，并支持将多份旧简历合并为根简历草稿（需用户确认）。
- **FR-004**: 根简历 MUST 支持模块化编辑、自由文本补充、AI 辅助整理、标签、技能-项目关联。
- **FR-005**: 系统 MUST 展示素材完整度提示（背景/职责/方案/结果/量化/技能证据等），且 MUST NOT 将其设计为强制评分门槛。
- **FR-006**: 用户 MUST 能够将派生流程中确认的补充事实同步回根简历（可选）。

#### 一键派生与岗位联动

- **FR-007**: 系统 MUST 提供一键派生入口（简历中心首页、派生列表上方、岗位详情、空状态引导）。
- **FR-008**: 一键派生的岗位来源 MUST 为求职追踪已登记岗位；列表 MUST 展示公司、岗位、状态、JD 完整性、是否已有派生、最近更新、是否推荐生成。
- **FR-009**: 无 JD 或 JD 为空的岗位 MUST NOT 允许开始一键派生，并 MUST 引导补充 JD。
- **FR-010**: 派生前用户 MUST 选择目标页数（仅 1/2/3）与模板（或系统推荐模板）。
- **FR-011**: 系统 MUST 对 JD 做结构化解析（岗位方向、职责、硬技能、加分项、行业、年限、高频/ATS 关键词、高中低优先级能力、根简历已有/缺失证据等），并以产品可读摘要展示。
- **FR-012**: 派生正文 MUST 仅使用根简历已有真实信息，或用户在流程中补充并确认的信息。
- **FR-013**: 系统 MUST NOT 编造经历、项目、公司、数据、技能掌握程度；MUST NOT 将 JD 要求直接写成用户已具备（除非有依据）。
- **FR-014**: 对 JD 高优先级能力，系统 MUST 优先展示相关项目/技能/成果；弱相关内容可压缩、合并或隐藏；无证据时 MUST 只生成「建议补充」，不得写入正式正文。
- **FR-015**: 系统 MUST 按 JD 重组结构（总结、优势、技能排序、项目优先级与侧重点），兼顾 ATS 与可读性，且 MUST NOT 关键词堆砌或改变事实含义。
- **FR-016**: 派生简历 MUST 绑定岗位、JD、生成时根简历版本、生成时间，并以独立快照保存。
- **FR-017**: 根简历后续修改 MUST NOT 自动改变已生成派生简历；系统 MAY 提示基于最新根简历重新生成（新版本）。

#### 严格页数控制

- **FR-018**: 目标页数是硬约束：最终导出 PDF 实际页数 MUST 严格等于用户选择的 1/2/3。
- **FR-019**: 未达到目标页数前，系统 MUST NOT 允许进入最终导出态。
- **FR-020**: 系统 MUST 在生成前设定内容预算与模块优先级（必须展示 / 可压缩 / 可隐藏）。
- **FR-021**: 系统 MUST 在真实渲染后检测实际页数；超页执行压缩策略，不足执行扩展策略；可多轮自动调整，并设最大自动调整轮次。
- **FR-022**: 超页压缩 MUST 按产品优先级执行（缩短总结 → 压缩弱相关项目/bullet → 合并技能 → 压缩教育/附加 → 更紧凑模板 → 适度行距/段距/字号且不低于可读性底线），并将移出内容放入「未采用素材」而非静默丢失。
- **FR-023**: 不足扩展 MUST NOT 编造事实；仅可扩展已有事实表达、展开分类/附加模块，或引导用户补充真实信息。
- **FR-024**: 自动调整仍无法严格满足时，系统 MUST 进入人工调整引导态（换模板、隐藏模块、减项目、补素材、改选页数等），且导出硬约束仍然生效。

#### 派生编辑器与 AI 建议

- **FR-025**: 派生编辑器 MUST 作为「JD 定向优化工作台」，同时呈现：当前派生简历、目标岗位、JD 匹配分析、AI 建议、页面状态、预览、导出入口。
- **FR-026**: 桌面端宜采用三栏（左结构导航 / 中编辑或预览 / 右 AI·匹配·页数）；小屏可折叠为 Tab。
- **FR-027**: AI 建议 MUST 包含：优先级、类型、对应位置、问题说明、对匹配影响、依据来源、可直接采纳/需补充/不建议写入、采纳前预览、采纳/忽略/稍后、可重新生成建议。
- **FR-028**: 建议类型至少覆盖：关键词补齐、项目强化、数据补充、内容删减、排序调整、表达优化、ATS 格式风险、页数控制、事实依据不足、关键词堆砌风险。
- **FR-029**: 所有 AI 对派生简历的修改 MUST 经用户确认后生效；MUST NOT 静默覆盖用户手动改动。
- **FR-030**: 用户 MUST 能查看内容取舍说明与（MVP 级）来源依据。

#### 补充信息闭环

- **FR-031**: 无事实依据时，Agent MUST 生成具体补充问题，MUST NOT 编造填空。
- **FR-032**: 用户补充后 MUST 可选择：仅当前派生 / 同步根简历 / 暂不使用；系统 MUST 保留补充来源记录。
- **FR-033**: 待用户确认内容 MUST 标记为待确认，且 MUST NOT 进入正式导出版。

#### 版本管理与导出

- **FR-034**: 派生简历 MUST 支持查看、编辑、复制、重新生成（新版本）、导出 PDF、删除、查看生成依据、查看历史版本；回滚与版本对比可作为 MVP 增强或紧随迭代（若工期紧，回滚/对比可标为 MVP+，但列表与重新生成必须有）。
- **FR-035**: 版本列表 MUST 展示：名称、绑定岗位/公司、目标页数、当前实际页数、生成/最后编辑时间、是否已导出、是否有待处理建议、是否来自最新根简历。
- **FR-036**: 用户 MUST 能从求职追踪岗位详情查看该岗位绑定的派生简历。
- **FR-037**: MVP MUST 支持 PDF 导出；Word 导出列为后续迭代。

#### 简历助手 Agent（产品约束）

- **FR-038**: Agent 输入可包括：根简历及其版本、当前派生、目标 JD、岗位信息、目标页数、模板、用户补充、历史采纳/忽略行为、页面渲染结果。
- **FR-039**: Agent 输出可包括：JD 解析、素材选择、取舍说明、派生正文、页数控制结果、匹配分析、关键词覆盖、优化建议、补充问题、风险提示、修改前后差异、来源依据。
- **FR-040**: 进入正式简历的事实 MUST 有来源，且来源仅限：根简历、用户上传旧简历、用户补充或用户确认信息。
- **FR-041**: Agent MUST 在必要位置提供「为什么这样优化」的解释。

### Key Entities

- **根简历**：用户级职业素材库；可很长；有版本概念供派生引用。
- **根简历素材项**：经历/项目/技能/成果等可引用单元；可带标签与完整度维度。
- **求职追踪岗位**：公司、岗位、状态、JD、更新时间等；派生的绑定对象。
- **JD 解析结果**：结构化要求与优先级、关键词、与根简历证据对照。
- **派生简历**：岗位定向投递快照；含目标页数、实际页数、模板、正文、绑定关系、生成元数据。
- **派生版本**：同岗位多次生成/编辑形成的版本序列。
- **未采用素材**：因页数或相关度未进入正文但仍可回看/恢复的素材集合。
- **AI 优化建议**：分级、分类型、可操作状态（待处理/已采纳/已忽略/稍后）。
- **用户补充事实**：问答产生的确认信息；可选同步根简历；有来源记录。
- **页数控制状态**：目标页数、实际页数、调整轮次、是否可导出、引导态原因。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新用户可在 10 分钟内完成根简历创建（空白或导入初稿）并保存至少 1 个完整项目模块。
- **SC-002**: 在已有根简历与含 JD 岗位的前提下，用户可在 5 分钟内完成一键派生并进入预览/编辑器（不含极端长任务排队）。
- **SC-003**: 在验收样例集上，导出 PDF 的实际页数与目标页数一致率达到 **100%**（不满足则不可导出，计为流程正确拦截而非失败导出）。
- **SC-004**: 抽检派生正文：无根简历/用户确认依据的「经历、项目、公司、量化数据、技能掌握断言」出现率为 **0**。
- **SC-005**: 无 JD 岗位尝试派生时，**100%** 被拦截并引导补充 JD。
- **SC-006**: 根简历修改后，已存在派生快照内容保持不变的比例为 **100%**（除非用户显式重新生成新版本）。
- **SC-007**: 派生编辑器中，用户对「可直接采纳」建议可在采纳前预览；未确认建议写入率为 **0**。
- **SC-008**: 岗位详情页可定位到该岗位全部绑定派生版本；用户能指出「本次投递对应哪一版」的任务完成率 ≥ **90%**（可用性走查）。
- **SC-009**: 页数不满足时进入引导态的案例中，用户能获得至少 2 条可执行下一步建议（换模板/减模块/补素材/改页数等）。
- **SC-010**: 核心路径（维护根简历 → 一键派生 → 页数达标导出）首次引导下任务完成率 ≥ **80%**（内部可用性测试）。

## Assumptions

1. 每位用户在 MVP 默认维护 **一份** 主根简历；多根简历/多身份素材库属后续能力。
2. 求职追踪模块已可登记岗位与 JD；本需求消费其岗位列表与详情，并回写「绑定派生」展示关系。
3. 简历模板与 PDF 渲染能力已存在或可复用；本需求新增「目标页数硬约束」与校准闭环。
4. MVP 导出以 **PDF** 为准做页数验收；Word 导出后置。
5. MVP 的 JD 匹配以清单式分析（覆盖/弱覆盖/未覆盖、缺口说明）为主；数值 Match Score / ATS 总分后置。
6. 「最大自动调整轮次」产品默认建议为有限次数（如 3–5 轮），具体数值在 plan 阶段标定；超限进人工引导。
7. 可读性底线（最小字号/行距）遵循现有设计规范；不得为凑页数破坏可读性。
8. 生成任务允许异步；用户可离开进度页，完成后在列表可见结果。
9. AI 合并多份旧简历、从上传解析初稿，产出均为「待用户确认」草稿。
10. 版本回滚与并排 diff 若影响 MVP 工期，可列为紧随的 MVP+，但不削弱「新版本不覆盖旧版本」与「重新生成」能力。

## Page Control Strategies *(product logic)*

### 生成前

- 按目标页数设内容预算与模块空间预估
- 标定必须展示 / 可压缩 / 可隐藏
- 对经历、项目、技能设优先级

### 生成中

- 控制项目数、bullet 数与长度、总结长度、技能密度、教育与附加篇幅

### 生成后

- 真实渲染 → 测页数 → 超页压缩 / 不足扩展 → 再渲染
- 达最大轮次仍失败 → 人工调整引导态
- 仅当实际页数 == 目标页数 才允许最终导出

## Future Advanced Capability Pool

以下为差异化高级能力池，**不纳入本 REQ MVP**，供后续路线图选用。

### 1. Career Vault 职业资产库

根简历升级为长期职业资产库：项目资产、技能证据链、可量化成果库、岗位方向标签、方向覆盖度、按目标岗反推缺口。

### 2. 简历证据链 / Provenance

每条派生内容可追溯：来自根简历哪一项目、对应 JD 哪条要求、技能证据项目、AI 改写 vs 用户确认事实。

### 3. JD-CV Fit Map 岗位匹配地图

非单一分数：JD 能力 × 根简历证据 × 派生是否展示 × 展示强度 × 缺口 × 推荐补充问题。

### 4. 多岗位批量派生

多选岗位批量生成独立快照；质量检查；低置信度标记；高价值岗位优先提醒。

### 5. 简历策略模式

保守真实 / 匹配增强 / ATS 优先 / 招聘官阅读优先 / 转岗解释 / 应届生潜力等。

### 6. 投递反馈反向优化

分析获面/无回应与能力表达、方向素材、版本表现的关联（需数据积累）。

### 7. 面试准备自动联动

基于派生 + JD 生成高频问、项目追问、自我介绍、STAR 草稿、技术清单、简历风险与可能质疑点。

### 8. Cover Letter / 自荐信联动

同一事实集生成求职信/私信；与派生一致；按平台调语气；避免重复堆砌。

### 9. 简历健康度雷达

匹配、关键词、可信度、数据完整度、项目说服力、ATS 友好、可读性、差异化、风险项等维度。

### 10. AI 追问式根简历补全

对话追问职责、结果、规模、指标、难点、解法、能力体现，整理进根简历。

### 11. 岗位方向素材覆盖度

选方向后展示覆盖度、缺证据、可迁移表达、优先补数据、技能需更多项目支撑。

### 12. 简历风险审查

夸大、依据不足、经历/时间线冲突、技能无证据、数据不可信、关键词堆砌、AI 腔、过度压缩、ATS 解析风险等。
