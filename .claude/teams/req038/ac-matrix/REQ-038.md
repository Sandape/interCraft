## Notes for Tester red-team

dev 自评的潜在反例 / 边界，留给 tester 接棒：

1. **DeepSeek JSON mode 假合规陷阱（L009）**：`response_format={"type":"json_object"}` 返回合法 JSON 但字段错。AC-010 是关键。
2. **mock 与生产 API 不一致**：旧代码可能仅在 mock 侧走 Pydantic、真实侧还在 regex。AC-009 用 method 名 grep 拦这一类。
3. **state mutation before validate**：FR-004 反例代码写法可能是 `state["x"] = raw; validated = schema.parse(raw)`，需检查顺序。
4. **fallback 静默吞错**：FR-007 反例是 `except: pass` 或 fallback 到 default object。必须 config 字段化。
5. **category 枚举漂移**：日志里 category 写成 `"SchemaInvalid"` / `"parse_fail"` / `"parse-fail"` 不统一。AC-008 应枚举全集。
6. **A2A 边界（deferred）**：FR-011 `input_schema` / `output_schema` 落到 US2，本轮不展开。
7. **mock fixture 退化**：mock 仅支持 `text` 字段、不接 schema，会让 e2e 失真。AC-009 + Fixtures 表共同约束。
8. **schema 校验副作用**：Pydantic v2 `model_validator` 自身抛错时不要误归 `schema_invalid` 而应归 `parse_fail` —— 与 tester 协商是否需要细化子分类。
9. **quota / timeout 与 schema 失败解耦**：rate limit 触发的 429 不应走 schema validator，必须上游拦截归 `quota`。AC-008 隐含此点。
10. **轮回使用旧 fixture 的回归风险**：旧 cycle 留下的 `mock_llm_*.json` 不一定覆盖 4 类；新增 / 替换时不要合并入历史 fixture 文件。
11. **mock by_scenario API 必须能拒绝 invalid schema name**（R11 强化）：`MockLLMClient.by_scenario("nonexistent_xyz")` 必须抛 `KeyError` 且错误信息含当前可用 fixture 列表（如 `"Available scenarios: malformed, missing, enum_violation, oob, quota, timeout"`），否则 dev 可能用 silent fallback 把 invalid name 兜成 `text` mock，破坏 e2e 真实性。验证命令：`cd backend && uv run pytest tests/test_mock_shares_structured_api.py::test_by_scenario_rejects_invalid -v`，断言 `pytest.raises(KeyError)` 且 `str(exc.value)` 包含 `"Available scenarios:"`。
